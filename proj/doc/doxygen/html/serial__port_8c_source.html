236  /*
237  The communication protocol is as follows:
238 
239  If bit 7 is up, it is a mouse packet (and it is expected that 4 bytes arrive)
240  If bit 7 isn't up, it is a scancode or END code (0x40).
241 
242  The composition of the mouse packet is changed:
243 
244  -The first byte has the following meaning:
245 
246  BIT(0) through BIT(4) -> the 5 most significant bits of the X position
247  BIT(5) -> if byte 3 is 0 (avoid queue pop unwanted events)
248  BIT(6) -> if byte 4 is 0 (avoid queue pop unwanted events)
249  BIT(7) -> if it is mouse info
250 
251  -The second byte has the following meaning:
252 
253  BIT(0) through BIT(4) -> the 5 most sigifnicant bits of the Y position
254  BIT(5) -> if bits 0 through 4 are all 0 (avoid queue pop unwanted events)
255  BIT(6) and BIT(7) -> unused
256 
257  -The third and fourth bytes represent the same:
258 
259  Byte 3 -> 8 least significant bits of the X position
260  Byte 4 -> 8 least significant bits of the Y position
261 
262  It must be remembered that the mouse packet is only sent when the lb is pressed (as it is only important to know
263  when the co-op player sent a magic blast), and that is assumed by the info receiver.
264 
265  The composition of a scancode is also changed, and is as follows:
266 
267  BIT(0) up -> A make code
268  BIT(1) up -> A break code
269  BIT(2) up -> D make code
270  BIT(3) up -> D break code
271  BITS 4 through 6 -> unused
272  BIT(7) -> it is a scancode
273 
274  When a mouse byte is first received, the following 3 packets are, by the queue logic, also from the packet, so they aren't checked.
275 
276  If the byte 0x40 == END is received at any given time, the game is finished.
277  */
